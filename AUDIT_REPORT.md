## 2. Почему организации не собираются

Обе диагностики согласны в главном: **технически парсер исправен, но в прод-БД ноль собранных организаций**. Историческая ошибка `Bound is incorrect → 0 карточек` уже устранена двумя фиксами, и они подтверждённо работают. Текущая причина нуля данных — **не парсер**, а адресат записи + отсутствие персистентности/автозапуска. Ниже причины по приоритету.

### BLOCKER — парсер пишет не в ту БД
**Доказательство.** Сегодняшние прогоны (2026-06-10, raw_8/9/10) успешны (по 5 карточек: ВкусВилл/Магнит/Чижик…, finalize «Готово»), но лог гласит «Старт парсера. БД: `c:\tmp\dash_test.db`» — `ext_runs.id=8,9,10` создались **в тестовой БД**. Путь берётся из env `DASHBOARD_DB` (подтверждено в коде: `parser/scrape_2gis.py:56` — `DB_PATH = Path(os.environ.get('DASHBOARD_DB', str(HERE.parent / 'dashboard.db')))`). В проде `ext_runs` содержит только id=2 (`excel-import`), `sqlite_sequence.ext_runs=7` — id 8/9/10 в проде даже не выделялись.
**Фикс.** Запускать боевой прогон с `DASHBOARD_DB`, указывающим на прод `dashboard.db` (или вовсе без env — тогда сработает дефолт `HERE.parent/dashboard.db`). Тестовая `c:\tmp\dash_test.db` не должна использоваться для прода.

### MAJOR — флапающий краш DOM.resolveNode (гонка/тайминг/антибот)
**Доказательство.** Логи 2026-05-07: на 4 прогонах **одной** категории (raw_4..7) — 3 фейла, 1 успех (~75% фейлов в той сессии). raw_5/raw_6: traceback в `perform_click → DOM.resolveNode` «No node with given id found», `returncode=0`, 0 карточек; raw_7 — успех на том же URL. Код это не ретраит: `scrape_url` при `returncode==0` и крахе просто возвращает `[]` (диагностика указывает `scrape_2gis.py:206`), а главный цикл (≈514) исключения не видит — категория молча даёт 0.
**Расхождение источников (важно).** Живой тест №1 сообщает: краха DOM.resolveNode **нет ни в одном логе**, все прогоны прошли. Статический анализ №2 показывает, что краш **реально воспроизводился** 2026-05-07 (3 из 4). Объяснение: живой тест шёл с `--max-records 5` за ~8–12 сек и просто «не успел» поймать флап; на полном объёме (403 категории) флап почти гарантированно ударит по части категорий. То есть «всё работает» из теста №1 — это удача малого прогона, а не доказательство стабильности.
**Фикс.** Добавить retry на категорию (2–3 попытки с экспоненциальной паузой), распознавать в `parser.log` признаки `DOM.resolveNode` и `Bound is incorrect` как retriable; увеличить `delay_between_clicks`/прогрев страницы; логировать категории, оставшиеся с 0, чтобы добивать вручную.

### MAJOR — отсутствует автозапуск (Task Scheduler)
**Доказательство.** В Windows Task Scheduler нет задачи для `scrape_2gis.py`: из 372 задач по ключам `2gis|scrape|parser|trc` совпала только встроенная Windows `MNO Metadata Parser`. Docstring обещает «еженедельно, суббота 03:00», но задача не создана.
**Фикс.** Создать задачу (python.exe + полный путь к скрипту, env на боевую БД, сб 03:00, наивысшие права). **Только после** устранения флапа, иначе ночные прогоны будут тихо падать.

### MAJOR — боевая БД пересоздаётся и затирает прогоны
**Доказательство.** `sqlite_sequence`: `ext_runs=7`, `ext_orgs=6531`, `ext_snapshots=6531`, при этом `ext_orgs.id` идёт 1..3264 (рестарт с 1), `ext_snapshots.id` — 3265..6528. Прогоны 1,3,4,5,6,7 реально выделялись (`begin_run` вставляет строку сразу), но исчезли. Excel-import даже с force удаляет **только** `source='excel-import'` (`external-import.ts:184-192`), оргов не трогает — значит затирался **весь файл БД**. `dashboard.db` в `.gitignore` («пересоздаётся на новой машине»).
**Фикс.** Вынести `dashboard.db` из логики «пересоздаётся на новой машине», ввести бэкап/миграцию вместо wipe; перед боевым прогоном убедиться, что работаем с постоянным файлом, который не будет затёрт следующим reseed.

### MINOR — длительность полного прогона / нет возобновляемости
**Доказательство.** 403 активных категории, `--delay` default 30 с между категориями, ~6–15 с парсинга на категорию, `timeout` одного `scrape_url`=1800 с. Оценка: 403 × ~40 с ≈ 4.5 ч, плюс ретраи. Укладывается в ночное окно, но без чекпойнтинга падение в середине требует полного перезапуска.
**Фикс.** Снизить delay до 10–15 с; сделать прогон возобновляемым (пропуск уже обработанных в этом run категорий).

### MINOR — хрупкость вендорного патча
**Доказательство.** `parser_2gis` импортируется из `parser/vendor-parser-2gis/` с патчем headless (`--window-size=1920,1080`), без которого возвращается `Bound is incorrect`. `pip install --upgrade parser-2gis` из PyPI затрёт патч.
**Фикс.** Закрепить установку из вендорной папки (`pip install -e parser/vendor-parser-2gis`), не апгрейдить из PyPI, задокументировать в README/requirements.

**Что НЕ является причиной нуля данных (оба источника согласны):** `normalize_search_url` (корректно стрипает `/filters/bound?m=...`, подтверждено в `parser/scrape_2gis.py:76-85`) и headless-патч `--window-size` — они исправны.
