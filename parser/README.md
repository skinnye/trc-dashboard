# Парсеры

В этой папке три парсера:

| Скрипт | Назначение | Куда пишет |
|---|---|---|
| `import_status_history.py` | Помесячная заполняемость помещений (Сдан / Не сдан) | `rent_status_history` |
| `import_movements.py`      | Съезды/заезды арендаторов (лист «4 съезд-заезд») | `tenant_movements` |
| `scrape_2gis.py`           | Парсинг карточек 2GIS (рейтинг, отзывы, координаты) | `ext_orgs`, `ext_snapshots` |

Источники для первых двух — годовые Excel-файлы план-факт со шары
`\\Acad-server\общие\02_Бухгалтерия\01_Бюджет\<год>\`.

## ⚙️ Автоматический запуск (рекомендуется)

Один раз настроите расписание через Windows Task Scheduler — дальше всё
будет крутиться само:

```powershell
# Правый клик → «Запустить от имени администратора»:
parser\setup_schedule.bat
```

После этого зарегистрируются две задачи:

| Задача | Когда | Что делает |
|---|---|---|
| `TRC_PlanFakt`      | каждый день 02:00 | импорт плана-факта (`run_all.bat`) |
| `TRC_2GIS_Scrape`   | суббота 03:00     | парсинг 2GIS (`run.bat`) |

Проверить, что задачи зарегистрировались:
```powershell
schtasks /Query /TN "TRC_PlanFakt"
schtasks /Query /TN "TRC_2GIS_Scrape"
```

Снять расписание:
```powershell
parser\unschedule.bat   # тоже от админа
```

Логи каждого автозапуска лежат в `parser\logs\`.

# Парсер 2GIS (детали)

Еженедельный сбор данных по категориям организаций для дашборда `/external`.

## Установка (один раз)

```powershell
cd parser
python -m venv venv          # опционально, но рекомендуется
venv\Scripts\activate

# Сторонний parser-2gis качаем как git-clone (vendor-) и ставим editable.
# В репо его не храним — он есть в .gitignore. Это позволяет легко
# подгонять под наш проект, не плодя форка.
git clone https://github.com/interlark/parser-2gis.git vendor-parser-2gis
pip install -e ./vendor-parser-2gis
pip install openpyxl
```

Парсер использует **установленный в системе Google Chrome** через Chrome DevTools Protocol (pychrome). Если Chrome ещё не стоит — поставьте https://www.google.com/chrome/.

## ⚠️ Важно: окно браузера будет открываться

Парсер **не работает в headless-режиме**: 2GIS Catalog API возвращает «Bound is incorrect» при viewport нулевого размера, который у безголового Chrome. Поэтому при каждом прогоне парсер открывает **видимое** окно Chrome и работает в нём.

Что это значит на практике:
- Если запускаете руками — на экране откроется окно, побежит, и закроется в конце. Можно свернуть.
- Если запускаете через Task Scheduler ночью — никого нет, окно просто открывается и закрывается, никому не мешает.
- Если параллельно вы пользуетесь компьютером — окно может перехватывать фокус. Лучше планировать прогон на нерабочее время.

## Ручной запуск

```powershell
python scrape_2gis.py                              # все активные категории (~403)
python scrape_2gis.py --limit 10                   # MVP-прогон на 10 категориях
python scrape_2gis.py --category 5                 # только одна категория
python scrape_2gis.py --delay 60                   # пауза 60 сек между категориями
python scrape_2gis.py --max-records 100            # лимит записей с одной категории
python scrape_2gis.py --keep-raw                   # сохранять JSON-выгрузки в logs/

# Smoke-тест (быстрый, ~10 сек):
python scrape_2gis.py --limit 1 --max-records 3 --keep-raw
```

## Время прогона

Парсер открывает каждую карточку и кликает по ней (так 2GIS отдаёт полные данные). Эмпирически:

- ~3 записи: **5–10 сек**
- ~50 записей: **2–4 мин**
- ~200 записей: **8–15 мин**

С 403 категориями за один прогон будет около **15–25 часов**. Это слишком много для одной ночи. Варианты:
- Разделить категории на 7 пакетов и парсить ежедневно (`--limit 60`)
- Ограничить `--max-records 100` — меньше каждой категории, прогон уложится в 5–8 часов
- Запускать только конкретные категории, важные для бизнеса

Лог пишется в `parser/logs/YYYY-MM-DD_HH-MM.log`. Параллельно выводится в консоль.

## Идемпотентность

При повторном запуске **дубликаты не создаются**:

- `ext_orgs` имеет UNIQUE на `dedupe_key = lower("категория|название|адрес")`. Уже виденные карточки — UPDATE'ятся (только `last_seen_at` и базовые поля); никаких лишних строк.
- В `ext_snapshots` пишется новая строка с текущим `run_id`. Это и есть «история» — её количество растёт линейно по числу прогонов, дубликатов в смысле «таже карточка дважды» не появляется.

Чтобы увидеть, что нового появилось/исчезло за прогон:

```sql
-- Появились в этом прогоне (первый раз увидели)
SELECT * FROM ext_orgs WHERE first_seen_at >= (
  SELECT started_at FROM ext_runs WHERE id = (SELECT MAX(id) FROM ext_runs WHERE status='ok')
);

-- Исчезли (раньше видели, в этом прогоне нет)
SELECT * FROM ext_orgs WHERE last_seen_at < (
  SELECT started_at FROM ext_runs WHERE id = (SELECT MAX(id) FROM ext_runs WHERE status='ok')
);
```

## Расписание (Windows Task Scheduler)

1. Открыть «Планировщик заданий».
2. «Создать задачу…».
3. Имя: `TRC dashboard · 2GIS scraper`.
4. Триггер: **еженедельно**, суббота, **03:00**.
5. Действие: «Запустить программу», обзор → `parser\run.bat`.
6. Условия: «Запускать, только если ПК подключён к сети».
7. Параметры: «Запускать с наивысшими правами» (нужно для Chromium).

Проверить, что отработало: посмотреть последний `parser/logs/*.log` или вкладку «Журнал изменений» в самом планировщике.

## URL-формат категорий

Excel из 2GIS экспорта содержит ссылки вида:
```
https://2gis.ru/ekaterinburg/search/Магазин/filters/bound?m=60.530548,56.788963/14
```

Параметр `?m=lng,lat/zoom` 2GIS Catalog API не принимает (отвечает «Bound is incorrect»). Поэтому скрипт автоматически срезает суффикс `/filters/bound?m=…` и парсит по всему городу. Локационная фильтрация делается потом — у каждой найденной карточки есть lat/lng, дашборд может ограничиваться расстоянием от ТРЦ Академический.

## Что делать, если 2GIS прислал капчу

`parser-2gis` ловит её и завершает прогон ошибкой. В `ext_runs` статус будет `error`, в логе — стектрейс. Решение:

1. Запустить руками `python scrape_2gis.py --limit 1 --keep-raw` — увидите, какие данные получили перед обвалом (`logs/raw_*.json` + `raw_*.parser.log`).
2. Увеличить `--delay` (например, до 120 сек) и попробовать снова.
3. Сократить охват: разделить категории на пакеты по 50 и запускать 2-3 раза в неделю.

## Удалить прогон, который пошёл не так

Если в БД появился прогон с битыми данными — просто стираем его run_id (FK не требует ничего):

```sql
DELETE FROM ext_snapshots WHERE run_id = ?;
DELETE FROM ext_runs WHERE id = ?;
```

Орги остаются — это история. UI всегда читает «последний успешный прогон».
